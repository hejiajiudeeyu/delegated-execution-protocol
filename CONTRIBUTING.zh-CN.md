# 为远程子代理协议做出贡献

感谢您参与远程子代理协议。

本仓库当前是一个 MVP 设计与实现早期，贡献目标是快速验证`远程子代理协议`闭环，并探讨第一个参考实现，而不是一次性做完整的生产系统。

## 1.贡献范围

欢迎以下类型贡献：
- 文档与接口完善规范（`docs/`）
- 买家/卖家参考实际与示例
- 平台最小服务（catalog/token/events/metrics）实现
- Playground交互演示与说明一致性修复
- 测试、可安装性、开发者体验改进

暂不接受：
- 与 MVP 边界无关的大规模重构

## 2. 开发流程

1.先在Issue中认领任务（或新建Issue说明问题）。
2. 新建分支：`codex/<scope>-<short-desc>`。
3.小步提交，保持PR可审查（建议< 400行净队列）。
4.提交PR，关联Issue，并填写变更说明和验收方式。
5. 至少 1 名维护者通过后合并审核。

## 3. 提交与 PR 规范

提交留言建议：
- `壮举：...`
- `修复：...`
- `文档：...`
- `重构：...`
- `测试：...`
- `家务：...`

PR 描述至少包含：
- 变更背景（为什么做）
- 变更内容（做了什么）
- 验证方式（如何证明有效）
- 风险与回滚点（如有）

## 4. 文档文档要求

涉及协议/接口/状态机的关系，必须同步更新：
- `docs/l0/architecture.md`
- `docs/l0/platform-api-v0.1.md`
- `docs/l0/integration-playbook.md`

如仅做探索性倡议，请先发ADR或Issue讨论，不直接修改主规范。

## 5.质量事业

合并前请确保：
- 范围范围聚焦单一目标
- 无明显破坏性变更（除非有明确升级说明）
- 示例 JSON 与文档字段一致
- 新增接口有错误提示

## 6.沟通与响应

- 建议先讨论再修改大项。
- 维护者目标响应：
  - 24小时内首次反馈
  - 72小时内给出治疗/修改建议

## 7. 首次贡献建议

第一次贡献优先从以下任务开始：
- 文档中的字段一致性检查
- Playground 与 API 文档文档修订
- 新增或修复一个小型示例模板

感谢你的贡献。
