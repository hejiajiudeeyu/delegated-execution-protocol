# 为远程热线协议做出贡献

感谢你参与远程热线协议。

本仓库当前仍处于 MVP 设计与实现早期，贡献目标是快速验证协议闭环，并推进首个参考实现，而不是一次性做成完整生产系统。

## 1. 贡献范围

欢迎以下类型的贡献：
- 文档与接口规范完善（`docs/`）
- Caller / Responder 参考实现与示例
- 平台最小服务（catalog、token、events、metrics）实现
- Playground 交互演示与文档一致性修复
- 测试、可安装性与开发者体验改进

暂不接受：
- 与 MVP 边界无关的大规模重构

## 2. 开发流程

1. 先在 Issue 中认领任务，或新建 Issue 说明问题。  
2. 新建分支：`codex/<scope>-<short-desc>`。  
3. 小步提交，保持 PR 可审查。  
4. 提交 PR，关联 Issue，并写清变更说明与验证方式。  
5. 至少 1 名维护者审核通过后再合并。  

## 3. 提交与 PR 规范

提交信息建议使用：
- `feat: ...`
- `fix: ...`
- `docs: ...`
- `refactor: ...`
- `test: ...`
- `chore: ...`

PR 描述至少包含：
- 变更背景
- 变更内容
- 验证方式
- 风险与回滚点（如有）

## 4. 文档要求

涉及协议、接口或状态机变更时，必须同步更新：
- `docs/current/spec/architecture.md`
- `docs/current/spec/platform-api-v0.1.md`
- `docs/current/guides/integration-playbook.md`

如果只是探索性提案，请先发 ADR 或 Issue 讨论，不直接改主规范。

## 5. 质量要求

合并前请确保：
- 变更范围聚焦单一目标
- 没有未声明的破坏性变更
- 示例 JSON 与文档字段一致
- 新增接口包含错误路径说明

## 6. 沟通与响应

- 大项变更建议先讨论再动手。  
- 维护者目标响应时效：
  - 24 小时内首次反馈
  - 72 小时内给出处理建议

## 7. 首次贡献建议

第一次贡献建议优先从以下任务开始：
- 文档字段一致性检查
- Playground 与 API 文档修订
- 新增或修复一个小型示例模板

感谢你的贡献。
